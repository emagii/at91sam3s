00389  \subsection Descriptors
00390  The USB specification 2.0 defines a set of descriptors used to give
00391  information about the device. Depending on the USB class implemented,
00392  different descriptors have to be used with varying values.
00393  
00394  In this example program, only a few descriptors are required. The device
00395  descriptor is always mandatory, so it will have to be defined. At least one
00396  configuration descriptor is required, so one is implemented. The described
00397  configuration must have at least one interface, so one more descriptor is
00398  needed. Finally, no string descriptors are used:
00399  
00400  \subsubsection  Device Descriptor
00401  The device descriptor used by this example looks like this:
00402  \code
00403  *  Device descriptor.
00404 const USBDeviceDescriptor usbDeviceDescriptor = {
00405 
00406     sizeof(USBDeviceDescriptor),
00407     USBGenericDescriptor_DEVICE,
00408     USBDeviceDescriptor_USB2_00,
00409     0, // No device class code
00410     0, // No device subclass code
00411     0, // No device protocol code
00412     BOARD_USB_ENDPOINTS_MAXPACKETSIZE(0),
00413     0x03EB, // Atmel vendor ID
00414     0x0001, // Product ID
00415     0x0001, // Product release 0.01
00416     0, // No manufacturer string descriptor
00417     0, // No product string descriptor
00418     0, // No serial number string descriptor
00419     1 // One possible configuration
00420 };
00421  \endcode
00422  The values are nothing special here. Note that the first three fields have the
00423  same data in them (unless using USB 1.1). It is also very common to define the
00424  class, subclass and protocol values at the interface level.
00425  
00426  - Note: The <i>vendor ID</i> value is provided by the USB-IF organization. The
00427  <i>product ID</i> is vendor-specific and can be assigned any value.
00428  
00429  \subsubsection Configuration & Interface
00430  When the configuration descriptor is requested by the host, via the
00431  GET_DESCRIPTOR command, the device must not only transmit this descriptor but
00432  also all the necessary interface and endpoint descriptors.
00433  
00434  In order to do that easily, a structure must be defined for holding all the
00435  information. This way, the data to send is contiguous, making the request
00436  much simpler to fulfill. In the current example, the configuration descriptor
00437  must be followed by the first interface descriptor. The following structure is
00438  declared for that:
00439  \code
00440 // Configuration descriptors with one interface.
00441 struct SimpleConfigurationDescriptors {
00442 
00443     USBConfigurationDescriptor configuration;
00444     USBInterfaceDescriptor interface;
00445 };
00446  \endcode
00447  Now, the actual descriptors can be declared:
00448  \code
00449 // Configuration descriptors.
00450 const struct SimpleConfigurationDescriptors configurationDescriptors = {
00451 
00452     // Configuration descriptor
00453     {
00454         sizeof(USBConfigurationDescriptor),
00455         USBGenericDescriptor_CONFIGURATION,
00456         sizeof(struct SimpleConfigurationDescriptors),
00457         0, // No interface in this configuration
00458         1, // This is configuration #1
00459         0, // No string descriptor for this configuration
00460         BOARD_USB_BMATTRIBUTES,
00461         USBConfigurationDescriptor_POWER(100)
00462     },
00463     // Interface descriptor
00464     {
00465         sizeof(USBInterfaceDescriptor),
00466         USBGenericDescriptor_INTERFACE,
00467         0, // This is interface #0
00468         0, // This is setting #0 for interface
00469         0, // Interface has no endpoint
00470         0, // No interface class code
00471         0, // No interface subclass code
00472         0, // No interface protocol code
00473         0, // No string descriptor
00474     }
00475 };
00476  \endcode
00477  Again, those are very generic values. For the interface descriptor, most of
00478  them are zeroed. This is because this example does not implement any
00479  functionality other than doing the USB enumeration.
00480

00482  The demonstration program is going to use the standard request handler

00487  \subsubsection  Descriptors List
00488  
00489  The \ref USBDDriver object needs a pointer to a list of descriptors. This is
00490  necessary to be able to answer the GET_DESCRIPTOR request. A
00491  \ref USBDDriverDescriptors can be used to do that.
00492  
00493  The actual descriptors list can be instantiated:
00494  \code
00495 // List of descriptors used by the device.
00496 const USBDDriverDescriptors usbdDriverDescriptors = {
00497 
00498     &usbDeviceDescriptor,
00499     (const USBConfigurationDescriptor *) &configurationDescriptors,
00500 #ifdef BOARD_USB_UDPHS
00501     0, // No full-speed device qualifier descriptor
00502     0, // No full-speed other speed configuration descriptor
00503     &usbDeviceDescriptor,
00504     (const USBConfigurationDescriptor *) &configurationDescriptors,
00505     0, // No high-speed device qualifier descriptor
00506     0, // No high-speed other speed configuration descriptor
00507 #else
00508     0, // No full-speed device qualifier descriptor
00509     0, // No full-speed other speed configuration descriptor
00510     0, // No high-speed device descriptor
00511     0, // No high-speed configuration descriptor
00512     0, // No high-speed device qualifier descriptor
00513     0, // No high-speed other speed configuration descriptor
00514 #endif
00515     0, // No string descriptor
00516     0  // No string descriptor
00517 };
00518  \endcode
00519  The core configuration descriptor, which is actually made up of the
00520  configuation descriptor and the first interface descriptor, has to be cast to
00521  the \ref USBConfigurationDescriptor type.

00523  \section Interrupt Service Routines
00524 
00525  \subsection USB Controller Interrupt
00526  The USB controller peripheral generates an interrupt when an event occurs.
00527  Since that event must be forwarded to the USBD_IrqHandler method, an
00528  interrupt service routine must be installed to do that.
00529  \code
00530  void ISR_Usb()
00531  {
00532     USBD_IrqHandler();
00533  }
00534  \endcode
00535  Currently the default USBDCallbacks_Initialized will initialize the
00536  interrupt service routine automatically:
00537  \code
00538 void USBDCallbacks_Initialized()
00539 {
00540     AIC_ConfigureIT(AT91C_ID_UDP, 0, USBD_IrqHandler);
00541     AIC_EnableIT(AT91C_ID_UDP);
00542 }
00543  \endcode
00544  So the only thing you should do to initialize the interrupt service is to call
00545  USBD_Init() in main().
00546  
00547  \subsection VBus PIO Interrupt
00548  The Vbus power line can be monitored (if a PIO pin is connected to it) by the
00549  user application to enable or disable the USB controller peripheral when the
00550  device is connected/disconnected. To do that, an interrupt must be programmed
00551  when the status of Vbus changes. The ISR should call the USBD_Connect() or
00552  USBD_Disconnect() function as follows:
00553  \code
00554 void ISR_Vbus(const Pin *pPin)
00555 {
00556     // Check current level on VBus
00557     if (PIO_Get(&pinVbus)) {
00558         USBD_Connect();
00559     }
00560     else {
00561         USBD_Disconnect();
00562     }
00563 }
00564  \endcode

00571  \subsection Init
00572  When an OS is not being used, the primary function that the <i>Init</i> callback
00573  must perform is interrupt handler installation. The previously defined ISR
00574  is thus configured and enabled:
00575  \code
00576 void USBDCallbacks_Initialized()
00577 {
00578     // Configure and enable the UDP interrupt
00579     AIC_ConfigureIT(AT91C_ID_UDP, 0, USBD_IrqHandler);
00580     AIC_EnableIT(AT91C_ID_UDP);
00581 }
00582  \endcode
00583  The default callback is defined in USBDCallbacks_Initialized.c.

00585  \subsection susp_rsm Suspend & Resume
00586  The Suspend callback is used by the \ref usbd_api "USBD API" to notify the device that it
00587  should enter low-power mode if required. The default callback is defined
00588  in USBDCallbacks_Suspended.c.
00589  \code
00590 void USBDCallbacks_Suspended(void)
00591 {
00592     // Turn off LEDs
00593     LED_Clear(USBD_LEDPOWER);
00594     LED_Clear(USBD_LEDUSB);
00595 }
00596  \endcode
00597  The Resume callback has to perform the reverse operation. The default callback
00598  is defined in USBDCallbacks_Resumed.c.
00599  \code
00600 void USBDCallbacks_Resumed(void)
00601 {
00602     // Initialize LEDs
00603     LED_Configure(USBD_LEDPOWER);
00604     LED_Set(USBD_LEDPOWER);
00605     LED_Configure(USBD_LEDUSB);
00606     LED_Clear(USBD_LEDUSB);
00607 }
00608  \endcode
00609  
00610  - Note: <i>It is not necessary to disable the USB controller logic (transceiver,
00611  clocks, peripheral) here. This is done directly by the USBD_IrqHandler
00612  function prior to triggering the callback. Typically, the callback must carry
00613  out the following operations:</i>
00614     - Disable the PLL
00615     - Switch to the slow 32 KHz clock
00616     - Turn off the clocks of used peripherals

00630  \section Main
00631  The Main function of the program is used for PIO and driver (Class and USB)
00632  initialization, software connections of the device (by using USBD_Connect),
00633  and implementation of the product functionality.
00634  
00635  In this case, the Main performs the first two steps. After that, since the
00636  enumeration is done through the event handler and the device does not do
00637  anything, it can simply enter an infinite loop:
00638  
00639  \code
00640 int main()
00641 {
00642     // If they are present, configure Vbus & Wake-up pins

00647     USBD_Init();
00648 
00649     // connect if needed
00650     VBUS_CONFIGURE();

00652     
00653     // Main loop
00654     while(1)
00655     {
00656         // Put USB class driver implementaion here
